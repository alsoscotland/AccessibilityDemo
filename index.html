Accessibility: The Basics of Screen Readers

  • Buttons

    Inaccessible Button

    I look like a button, but I'm not a button.
    If someone is directed to click the "button", the screen readers will not add this element to their list of buttons.
    Markup <div class="button" id="clickable_div"> I look like a button, but I'm not a button. </div>

    Problem:

    Screen readers give users helpful list-menus of certain semantic html elements. If an element which looks & behaves like a button, but is not semantically a <button> html element, it is much more difficult for a visually impaired user to locate the control

    Accessible Button

    Markup <button class="button" id="clickable_button"> I am a REAL Button </button>

    Solution:

    If it's a button, use a <button>!, they are semantically correct, and they work great for all users

  • Image Examples

    These images do not have alt-tags. So you hear the actual image file name

    Additionally, this Boba-Fett image in completely irrelevant. It is of no use to a blind user and should not be read

    Markup <div class="button" id="clickable_div"> <img src="img/aria_html5.png" /> <img src="img/bobafett.jpeg" /> </div>

    Problem:

    Images must have alt attribute tags. Even images we want ignored need one, just make its value an empty string

    These first of these two images has an alt-tag with a contextual text value. Screen Readers will read this information when they encounter the image.

    The Boba-Fett image, being irrelevant, has its alt-tag set to be an empty string. This will make screen reader software ignore the image altogether

    Markup <img src="img/aria_html5.png" alt="HTML5 ARIA Logo" /> <img src="img/bobafett.jpeg" alt="" />

    Solution:

    If your image is important in order to better understand the page content, use an alt attribute to describe your image. If not, use an alt with an empty string to tell screen readers to ignore it.

    Solution:

    By coupling <input> tags with <label> tags, screen readers will read cues to a visually impaired user when they focus the input field. Additionally, some readers support the HTML5 placeholder attribute which you can use to provide additional information. When displaying errors, adding an aria-invalid="true" attribute to the input will let the user know the field is in error.

    Markup <table> <tr> <td>Name</td> <td>Team</td> <td>Score</td> <td>Rank</td> </tr> <tr> <td>Dave</td> <td>The Champs</td> <td>10,800</td> <td>1</td> </tr> <tr> <td>Steve</td> <td>The Nerds</td> <td>600</td> <td>4</td> </tr> <tr> <td>Louise</td> <td>The Nerds</td> <td>650</td> <td>3</td> </tr> <tr> <td>Linda</td> <td>The Champs</td> <td>900</td> <td>2</td> </tr> </table>

    Problem:

    Without a caption tag and heading (TH tags), it is very easy for a user to get completelely lost in the data. Headings are repeatedly read by screen-readers as the user navigates, making the data much easier to understand.

    Markup <table> <caption> Pac-Man High Scores For The Week </caption> <tr> <th>Name</th> <th>Team</th> <th>Score</th> <th>Rank</th> </tr> <tr> <th>Dave</th> <td>The Champs</td> <td>10,800</td> <td>1</td> </tr> <tr> <th>Steve</th> <td>The Nerds</td> <td>600</td> <td>4</td> </tr> <tr> <th>Louise</th> <td>The Nerds</td> <td>650</td> <td>3</td> </tr> <tr> <th>Linda</th> <td>The Champs</td> <td>900</td> <td>2</td> </tr> </table>

    Solution:

    A <caption&rt;

    tag will be read to a visually impaired user when they navigate to your table. In addition, when you label column headers and row headers with <th&rt; tags, readers will repeat the respective headings when reading data from the table. This is crucial as it is easy to lose context when browsing through many data cells.

Screen Reader Help & Resources

The industry standard screen reader for PC is called JAWS. It has roughly 90% of the market share and is the standard program that companies would like us to test with.

When using JAWS it is important to start the program prior to opening a browser.

As Mac users & developers, we have access to a great screen reader called Voiceover which is included as a part of the OSX operating system. Voiceover can be enabled with the shortcut key command-F5 or via the System Preferences menus as pictured below.


Screen Readers are very complex Applications with large amounts of keyboard shortcuts to learn. Here are a few links to some help on getting started.

A few very useful Voiceover commands

  • The "voiceover keystroke", denoted as VO is the combination of
    Control + Option
  • To toggle voiceover on and off
    Command-F5
  • You can stop voiceover from speaking at any time with
    Control
  • To open the voiceover help menu
    VO + H
  • To open the voiceover commands help menu
    VO + HH
  • To open the Voiceover speech attributes menu
    VO + Command + left arrow
